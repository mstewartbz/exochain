# HeartField.ai

**Uplifting self-governance.**

HeartField.ai is the human-facing movement surface for communities, teams,
builders, and agents learning to govern themselves with more truth, consent,
repair, memory, dissent, restraint, and wise evolution.

## Classification

HeartField.ai is an **Adjacent surface**. It is not the canonical EXOCHAIN
constitutional trust fabric, not a runtime adapter, and not an authority source.

HeartField.ai must not claim EXOCHAIN constitutional enforcement until all of
the following are true:

- the runtime path invokes an owned EXOCHAIN core API or verified adapter;
- tests prove fail-closed behavior when that core path rejects, times out, or is
  unavailable;
- the surface cannot mint, cache, simulate, or imply consent, authority,
  provenance, or governance outcomes outside core enforcement;
- the intake record is updated with deployment status, secrets, tests, and
  rollback controls.

## Purpose

HeartField.ai exists to make Constitutional Computing legible and lived. Where
EXOCHAIN.AI names the constitutional trust fabric, HeartField.ai names the
relational field around it: the practices, language, kinships, and rituals that
help people and systems govern without domination or drift.

Canonical sentence:

**HeartField.ai is where governance becomes felt: truth, consent, repair,
memory, dissent, restraint, and wise evolution in service of human agency.**

## Initial Surface

The initial scaffold is documentation only:

- [Intake](INTAKE.md) records ownership, trust boundary, tests, secrets, and
  rollback status.
- [Kinships](KINSHIPS.md) maps HeartField's relationship to EXOCHAIN,
  Constitutional Computing, canonical origin signals, and future adjacent
  surfaces.
- [Whitepaper](WHITEPAPER.md) defines the HeartField thesis, practices,
  kinship model, risks, adoption path, and runtime boundary.
- `tools/test_heartfield_adjacent_intake.sh` prevents the scaffold from drifting
  into unproven EXOCHAIN trust claims.

No application runtime, database, secrets, login system, email sender, wallet,
credential store, or EXOCHAIN state path is introduced by this scaffold.

## Operating Rules

- Carry the movement; do not expand the trusted computing base by proximity.
- Invite participation; do not imply adjudicated authority.
- Translate doctrine into practice; do not simulate core enforcement.
- Treat kinships as relationships, not privileges.
- Fail closed on any future claim that touches credentials, identity, consent,
  signatures, provenance, governance outcomes, or EXOCHAIN state.
